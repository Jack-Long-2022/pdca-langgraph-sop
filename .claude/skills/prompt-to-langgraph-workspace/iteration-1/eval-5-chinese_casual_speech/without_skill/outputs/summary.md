# 口语化中文处理总结报告

## 任务概述

**用户原始输入（口语化中文）**:
> "那个，我想做一个自动写报告的流程。首先呢，用户输入一个主题，然后系统去搜索相关的资料，搜完之后呢，把结果整理一下，然后让AI分析生成一个报告大纲。大纲生成之后，如果用户觉得不满意，可以让AI重新生成，最多重试3次。大纲确定后，再根据大纲写完整的报告内容。"

## 处理过程分析

### 1. 输入预处理阶段

**识别到的口语特征**:
- **填充词**: "那个"、"首先呢"、"搜完之后呢"、"然后"
- **停顿标记**: 多个"然后"连接
- **口语连接词**: "首先呢"、"...之后呢"

**清洗后的核心内容**:
```
用户输入主题 → 搜索资料 → 整理结果 → 生成报告大纲 →
用户确认（如果不满意则重新生成，最多重试3次）→
根据大纲写完整报告
```

### 2. 语义结构识别

**顺序关系识别** (关键词: 首先、然后、之后、完成后):
- ✅ "首先呢，用户输入" → 输入节点
- ✅ "然后系统去搜索" → 搜索节点
- ✅ "搜完之后呢，把结果整理" → 整理节点
- ✅ "然后让AI分析生成大纲" → 生成大纲节点
- ✅ "大纲确定后，再根据大纲写" → 生成报告节点

**条件关系识别** (关键词: 如果、则、否则):
- ✅ "如果用户觉得不满意" → 条件分支起点
- ✅ "可以让AI重新生成" → 循环返回路径
- ✅ "最多重试3次" → 循环终止条件

### 3. 节点推断结果

**显式节点** (来自直接描述):
1. 搜索资料 - 显式提及 "系统去搜索相关的资料"
2. 整理搜索结果 - 显式提及 "把结果整理一下"
3. 生成报告大纲 - 显式提及 "让AI分析生成一个报告大纲"
4. 生成完整报告 - 显式提及 "根据大纲写完整的报告内容"

**隐式节点** (基于逻辑推断):
1. 输入准备节点 - 任何工作流都需要入口点
2. 用户确认审核节点 - "如果用户觉得不满意" 需要审核环节
3. 输出结果节点 - 工作流需要明确的出口点

### 4. 边关系推断

**顺序边** (5条):
- 输入准备 → 搜索资料
- 搜索资料 → 整理结果
- 整理结果 → 生成大纲
- 生成大纲 → 用户审核
- 生成完整报告 → 输出结果

**条件边** (2条):
- 用户审核 → 生成大纲 (不满意时循环，条件: `!user_approval && retry_count < 3`)
- 用户审核 → 生成报告 (满意时继续，条件: `user_approval == true`)

### 5. 状态推断

**输入状态**:
- `topic` - 用户输入的主题

**中间状态**:
- `search_results` - 搜索结果
- `organized_results` - 整理后的结果
- `outline` - 报告大纲

**控制状态**:
- `user_approval` - 用户是否批准
- `retry_count` - 重试计数器

**输出状态**:
- `final_report` - 最终报告
