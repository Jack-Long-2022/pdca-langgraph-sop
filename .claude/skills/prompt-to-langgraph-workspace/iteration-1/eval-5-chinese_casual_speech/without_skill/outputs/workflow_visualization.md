# 自动报告生成工作流 - 可视化说明

## 工作流概述

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          自动报告生成工作流                                    │
│  输入主题 → 搜索资料 → 整理结果 → 生成大纲 → [用户审核] → 生成完整报告 → 输出    │
│                                        ↓                                      │
│                                    (不满意时重试，最多3次)                     │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 节点详细说明

### 1. 输入准备 (node_control_001)
- **类型**: 控制节点 - 输入节点
- **功能**: 接收并验证用户输入的报告主题
- **输入**: raw_input (用户原始输入)
- **输出**: topic (验证后的主题)
- **验证规则**:
  - 主题不能为空
  - 主题至少5个字符

### 2. 搜索资料 (node_tool_001)
- **类型**: 工具节点 - 搜索
- **功能**: 根据主题执行网络搜索
- **输入**: topic, max_results (默认10)
- **输出**: search_results (搜索结果数组)
- **配置**: 30秒超时，失败重试2次

### 3. 整理搜索结果 (node_thinking_001)
- **类型**: 思维节点 - 总结
- **功能**: 去重、排序、分类、摘要生成
- **输入**: search_results
- **输出**: organized_results (整理后的结果对象)
- **AI配置**: temperature=0.3, max_tokens=2000

### 4. 生成报告大纲 (node_thinking_002)
- **类型**: 思维节点 - 生成
- **功能**: 基于整理结果生成结构化大纲
- **输入**: topic, organized_results
- **输出**: outline (包含章节结构和要点)
- **AI配置**: temperature=0.5, max_tokens=1500
- **输出要求**: 3-7个主要章节，每章节2-5个要点

### 5. 用户确认审核 (node_control_002)
- **类型**: 控制节点 - 人工审核
- **功能**: 等待用户审核大纲
- **输入**: outline, retry_count
- **输出**: user_approval, retry_count, feedback (可选)
- **配置**: 最多重试3次，10分钟超时

### 6. 生成完整报告 (node_thinking_003)
- **类型**: 思维节点 - 生成
- **功能**: 根据确认的大纲生成完整报告
- **输入**: topic, outline, organized_results
- **输出**: final_report (Markdown格式)
- **AI配置**: temperature=0.6, max_tokens=8000
- **输出要求**: 2000-5000字，专业逻辑清晰

### 7. 输出结果 (node_control_003)
- **类型**: 控制节点 - 输出节点
- **功能**: 格式化并输出最终结果
- **输入**: final_report, outline, topic
- **输出**: result (包含完整信息的对象)

## 边关系说明

### 顺序边 (Sequential)
1. **输入准备 → 搜索资料**: 验证通过后开始搜索
2. **搜索资料 → 整理结果**: 搜索完成后整理结果
3. **整理结果 → 生成大纲**: 整理完成后生成大纲
4. **生成大纲 → 用户审核**: 大纲生成后等待用户审核
5. **生成完整报告 → 输出结果**: 报告生成后输出

### 条件边 (Conditional)
1. **用户审核 → 生成大纲**: 当用户不满意且重试次数<3时
   - 条件: `!user_approval && retry_count < 3`
   - 行为: 递增重试计数，返回大纲生成

2. **用户审核 → 生成完整报告**: 当用户批准大纲时
   - 条件: `user_approval == true`
   - 行为: 继续执行报告生成

## 状态流转图

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                              状态流转                                          │
└──────────────────────────────────────────────────────────────────────────────┘

topic (input)
   │
   ├─→ search_results (intermediate)
   │     │
   │     └─→ organized_results (intermediate)
   │           │
   │           └─→ outline (intermediate)
   │                 │
   │                 ├─→ user_approval (control)
   │                 │     │
   │                 │     ├─ false → retry_count++ → [loop back]
   │                 │     │
   │                 │     └─ true → final_report (output)
   │
   └─→ result (output)
```

## 循环机制

**大纲生成循环**:
```
生成报告大纲 → 用户审核
     ↑              │
     │              └─→ 不满意 (user_approval=false)
     │                   且 retry_count < 3
     │
     └─→ 重新生成 (使用之前的反馈)
```

**终止条件**:
- 用户批准大纲 (user_approval=true)
- 达到最大重试次数 (retry_count >= 3)

## 配置参数

### 全局配置
- **超时时间**: 600000ms (10分钟)
- **最大重试次数**: 3
- **检查点启用**: 是
- **日志启用**: 是
- **执行模式**: 顺序执行
- **错误处理**: 遇错继续，记录日志

### 关键参数
| 参数 | 值 | 说明 |
|------|-----|------|
| max_search_results | 10 | 搜索结果最大数量 |
| search_timeout | 30000ms | 搜索操作超时 |
| approval_timeout | 600000ms | 用户审核超时 |
| max_outline_retries | 3 | 大纲最大重试次数 |
| report_min_length | 2000字 | 报告最小长度 |
| report_max_length | 5000字 | 报告最大长度 |

## 交互示例

### 成功流程
```
用户输入: "人工智能在教育领域的应用"
  ↓
系统搜索: 找到15篇相关文章
  ↓
整理结果: 去重后保留10篇，分为4个主题
  ↓
生成大纲: 创建5个章节，共18个要点
  ↓
用户审核: 批准 ✓
  ↓
生成报告: 生成3500字的完整报告
  ↓
输出结果: 返回报告、大纲、元数据
```

### 重试流程
```
生成大纲: 创建5个章节
  ↓
用户审核: 拒绝 ✗，反馈"缺少伦理考量部分"
  ↓ (retry_count: 0→1)
重新生成: 调整大纲，增加伦理章节
  ↓
用户审核: 拒绝 ✗，反馈"案例分析不够"
  ↓ (retry_count: 1→2)
重新生成: 补充案例分析
  ↓
用户审核: 批准 ✓
  ↓
继续生成报告...
```

## 潜在改进点

基于分析中的 ambiguities，以下方面可能需要进一步明确：

1. **搜索数据源**: 当前假设使用通用搜索，可能需要添加学术数据库或内部知识库
2. **整理标准**: 当前假设使用去重+排序+摘要，可能需要更精细的分类规则
3. **用户反馈机制**: 当前假设二元选择，可能需要评分系统或直接编辑功能
4. **重试上限行为**: 当前需要在达到3次重试后明确处理策略（接受或终止）
